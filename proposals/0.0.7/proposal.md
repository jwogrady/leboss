# LEBOSS Proposal 0.0.7

**Status:** Proposal
**Branch:** proposal/0.0.7
**Date:** 2026-03-09
**Builds on:** proposal/0.0.6

---

## Summary

This proposal introduces the **Data Portability Protocol** — the behavioral guarantees ensuring that a governing entity can export its complete operational data environment and migrate it to compatible infrastructure.

From the first version of LEBOSS, the doctrine has been unambiguous: operational data belongs to the business that generates it, not to the platform that stores it. Prior proposals have built the governance layer — rules, objects, authorization protocols, integration controls, and audit coverage. This proposal adds the enforcement layer: the system behaviors that make data ownership practically exercisable, not merely asserted.

Without enforceable portability, a governing entity that owns its data on paper may find that ownership unexercisable in practice — their data locked in proprietary formats, incomplete on export, or dependent on the continued goodwill of a vendor for access. The Data Portability Protocol eliminates those failure modes.

---

## Motivation

The LEBOSS normative rule register includes portability requirements: `LEBOSS-CONT-1` through `LEBOSS-CONT-4` address continuity and transition obligations, and the conformance section of the standard requires that data portability be a guaranteed property, not an optional feature. What the standard does not currently define is *how* that guarantee is operationalized — what a governing entity must be able to request, what a system must provide, and what constraints apply to the export process.

This gap has real consequences:

- A service provider can claim portability compliance while producing incomplete exports that omit historical records, configuration state, or integration metadata.
- A governing entity has no normative basis to require a specific export scope — only the general assertion that their data must be accessible.
- There is no defined audit requirement for export events, creating a gap in the governance record at precisely the moment when data is most at risk of unauthorized handling.

This proposal defines the behavioral requirements that close these gaps. It answers: what must be exported, who may request it, how completeness is verified, what format constraints apply, and what audit record the export must produce.

## Rationale

### Why now — after the audit protocol?

Portability requires audit coverage. An export event that produces no audit record creates a governance blind spot at the highest-risk moment in the data lifecycle. By establishing the Audit Record Collection Protocol in 0.0.6 before defining portability requirements in 0.0.7, the export audit requirements in this proposal are grounded in an already-established behavioral framework rather than asserting requirements against an undefined foundation.

### Why not mandate a specific export format?

The standard is implementation-neutral by design. Mandating a specific format — JSON, CSV, SQL dump, a custom schema — would constrain implementations without improving governance outcomes. The governance requirement is that the format be machine-readable, documented, and reasonably parseable by independent systems. What format satisfies those criteria is an implementation decision.

A future proposal may define a LEBOSS Portability Format as a recommended standard, but this proposal establishes the behavioral requirements that any format must satisfy.

### Why include audit records in the export scope?

Audit records are operational data in the governance sense — they are records generated by the business's operational environment. An export that omits audit records delivers incomplete operational history. More importantly, the governing entity's ability to verify what happened to their data after leaving a vendor depends on having a complete audit record in their custody. An export without audit records is a portability escape for the vendor.

## Backward Compatibility

No existing normative rules are modified. No existing document sections are removed or rewritten.

Prior portability-related normative rules (`LEBOSS-CONT-1` through `LEBOSS-CONT-4`) gain behavioral context from this protocol but are not modified.

---

## Implementation Considerations

The protocol specifies required behavioral properties. It does not specify:

- The technical format of export packages
- The delivery mechanism (download, object storage, physical transfer)
- The import mechanism or compatibility requirements for receiving systems
- Storage or compression requirements for export artifacts

These are implementation decisions. The requirement is that the behavioral outcome — complete export, authority verification, audit coverage, format neutrality — is achieved.

---

## Deferred to 0.0.8 and Beyond

- LEBOSS Portability Format specification (recommended canonical format for export packages)
- Cross-environment import and environment reconstitution protocols
- Succession and Ownership Transfer Protocol (GAP-5)
- Partial export use cases (e.g., single brand entity, single data category)

## Feedback Requested

**1. Export scope completeness**
Should the protocol define a formal hierarchy of export completeness levels (e.g., full environment, brand entity, data category) — or should completeness requirements apply only to full-environment exports?

**2. Response time requirements**
Should the protocol require that export requests be fulfilled within a defined time window, or is "upon request" sufficient at this stage?

**3. Ongoing portability vs. one-time export**
Should the protocol address continuous data access (e.g., live data feeds, ongoing sync) as a portability mechanism, or only export-on-demand?

**4. Format documentation requirements**
What constitutes sufficient format documentation for the export neutrality requirement? Should the protocol require that documentation be provided to the governing entity at the time of export, or is prior disclosure sufficient?
